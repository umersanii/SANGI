# SANGI v1 — Prototype Simplification Plan

## Context
SANGI currently spans ~6,500 LOC across ESP32 firmware, a Raspberry Pi service, a PC monitor, and a web dashboard — all glued together through AWS IoT Core mTLS. That's a lot of moving parts for what is fundamentally a cute desk robot. The goal: strip it down to a fun, self-contained gadget someone can pick up and enjoy immediately, no cloud accounts or extra devices required.

---

## What to CUT

### 1. AWS IoT Core / MQTT cloud layer
**Why**: Requires AWS account, certificate provisioning, IAM policies, endpoint configuration. Massive setup friction for a prototype.

### 2. Raspberry Pi notification service (`pi-setup/`)
**Why**: Requires a second always-on device. GitHub polling, Discord selfbot, and D-Bus monitor add infra complexity without being core to the "fun gadget" experience.

### 3. PC workspace monitor (`PC-setup/`)
**Why**: Third device in the chain. Cool concept but not essential for a prototype.

### 4. Web dashboard (`Frontend/sagni/`)
**Why**: Depends entirely on the AWS IoT MQTT broker. Replace with something simpler (see below).

### 5. Complex data-display emotions
- **GITHUB_STATS** (204 LOC, 131 frames) — needs live data from Pi
- **NOTIFICATION** (131 LOC) — scrolling text needs MQTT payload
- **CODING** (56 LOC) — needs PC workspace monitor

### 6. Firmware networking layer
- `network.cpp`, `network.h`, `mqtt_manager.cpp`, `mqtt_manager.h` (~810 LOC)
- `notification_queue.cpp/.h`, `github_data.cpp/.h` (~270 LOC)
- WiFiClientSecure, PubSubClient, ArduinoJson dependencies

**Total cut: ~3,500+ LOC across all layers, 3 devices → 1 device**

---

## What to KEEP

### Core charm (the whole product)
- **Animated face** — 12 emotions (IDLE, BLINK, HAPPY, SAD, ANGRY, EXCITED, SLEEPY, CONFUSED, THINKING, LOVE, SURPRISED, DEAD, MUSIC)
- **ICanvas + EmotionRegistry** — clean architecture, already refactored
- **Blink transition system** — smooth emotion switching
- **Touch input** — physical interaction
- **Speaker beeps** — satisfying audio feedback
- **Battery monitoring** — useful for portable gadget
- **Offline autonomous mode** — already exists, becomes the default mode

---

## What to ADD

### 1. BLE control (replaces AWS IoT + web dashboard)
- ESP32-C3 has built-in BLE
- Free phone apps (nRF Connect, LightBlue) can send commands immediately
- One BLE characteristic for emotion ID — phone sends `0x03`, robot becomes SAD
- Future: simple companion app (React Native / Flutter)
- **~150 LOC** using NimBLE-Arduino library

### 2. Autonomous personality (makes it a pet, not a display)
- **Mood drift**: slowly shift emotions based on time of day (energetic morning → sleepy evening)
- **Attention seeking**: if no touch for 15 min, cycle through SAD → CONFUSED → ANGRY
- **Touch response**: different emotions based on touch patterns (tap = HAPPY, long press = LOVE, double tap = EXCITED)
- **Random micro-expressions**: occasional blink patterns, small reactions
- **~200 LOC** in a new `personality.cpp`

### 3. Captive portal WiFi config (optional, for future OTA)
- ESP32 hosts its own WiFi AP + config page
- No external server needed
- Sets up for future OTA firmware updates
- Could serve a tiny emotion-control page directly from ESP32
- **~100 LOC** using ESP32 WebServer

## Resulting Prototype

| Aspect | Before | After |
|--------|--------|-------|
| Devices needed | 3 (ESP32 + Pi + PC) | 1 (ESP32 only) |
| Cloud accounts | AWS IoT Core | None |
| Setup time | ~1 hour (certs, configs) | Flash & go |
| Emotions | 16 (3 need external data) | 12 (all self-contained) |
| Remote control | Web dashboard via cloud | BLE from phone |
| Personality | Reactive only | Autonomous + reactive |
| ESP32 firmware LOC | ~4,200 | ~2,500 |
| Total project LOC | ~6,500 | ~2,700 |

---

## Implementation order
1. Strip networking (delete MQTT/WiFi/network files, remove from main.cpp)
2. Remove data-dependent emotions (GITHUB_STATS, NOTIFICATION, CODING)
3. Add BLE service for emotion control
4. Add personality engine (mood drift, attention seeking, touch patterns)
5. Optional: captive portal for config/OTA
